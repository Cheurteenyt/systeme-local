# Operator-evidence session lifecycle

Status: normative private Rust lifecycle contract, B1.1

## Purpose

This document defines the in-memory lifecycle invariants for future operator-evidence custody
sessions. It does not authorize filesystem access, raw evidence ingestion, sanitizer execution,
retention or deletion.

The B0 wire protocol remains unchanged and still exposes only `describe_contract`.

## Authority

Rust owns the private session state, transition validation, monotonic revision and transition
commitment.

Python remains authoritative for observation planning, source compatibility, freshness, public
provider models, existing public digest domains, bundle compilation and readiness interpretation.

## Session identifier

A session identifier is opaque and has this exact syntax:

```text
ses_[0-9a-f]{32}
```

The identifier:

- contains 128 bits represented as lowercase hexadecimal text;
- cannot contain a path separator, endpoint, workspace label or arbitrary operator text;
- is not generated by B1.1;
- is never interpreted as a filesystem path.

Random generation and persistence belong to a later custody lot.

## States

The exact states are:

```text
created
collecting
sealed
retained
aborted
expired
disposed
```

`disposed` is terminal.

## Actions

The exact actions are:

```text
begin_collection
seal
retain
abort
expire
dispose
```

## Authorized transition graph

```text
created -> collecting | aborted | expired
collecting -> sealed | aborted | expired
sealed -> retained | disposed
retained -> disposed
aborted -> disposed
expired -> disposed
disposed -> <none>
```

Every state/action pair absent from this graph fails closed.

A failed transition preserves the session state and revision.

## Revision

A new session starts at revision `0`.

Every successful transition:

1. validates the edge;
2. computes `revision + 1` with checked arithmetic;
3. creates the transition receipt;
4. commits the new state and revision.

Revision overflow fails closed without mutation.

## Transition receipt

A successful transition returns exactly:

```text
session_id
prior_state
action
resulting_state
revision
transition_sha256
```

The receipt contains no path, endpoint, raw evidence, timestamp, environment value, credential,
secret, token or provider payload.

The receipt is not a disposition receipt and does not claim logical deletion or physical erasure.

## Private transition commitment

The domain is the exact byte sequence:

```text
systeme-local:operator-evidence-session-transition:v1\x00
```

The commitment is:

```text
SHA-256(
  domain ||
  len64(session_id_utf8)       || session_id_utf8 ||
  len64(prior_state_utf8)      || prior_state_utf8 ||
  len64(action_utf8)           || action_utf8 ||
  len64(resulting_state_utf8)  || resulting_state_utf8 ||
  len64(revision_utf8)         || revision_utf8
)
```

Lengths are unsigned 64-bit big-endian integers. `revision_utf8` is the base-10 ASCII rendering of
the unsigned revision with no leading zero.

This private domain is separate from:

- the B0 contract commitment;
- source and sanitized evidence commitments planned for later B1 lots;
- every existing public provider digest domain.

## Failure disclosure

A transition error contains only:

```text
prior_state
action
```

The Rust error type distinguishes an illegal edge from revision overflow without including the
session identifier or any operator-controlled value.

## B1.1 non-goals

B1.1 implements no:

- protocol operation;
- protocol-version change;
- filesystem or network I/O;
- temporary directory;
- lock file;
- source path validation;
- file-type inspection;
- raw-byte custody;
- sanitizer;
- retention policy;
- disposition receipt;
- logical deletion;
- physical-erasure claim;
- Python model change;
- public provider change.

## Next gate

B1.2 may add canonical path and file-type validation plus bounded streaming against synthetic
temporary files. It must preserve this lifecycle contract and remain path-free and secret-free at
the Python boundary.
